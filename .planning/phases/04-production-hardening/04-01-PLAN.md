<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Internet → Express (endpoint público) | Railway health check poller e qualquer scanner externo chamam `/health` sem autenticação |
| Internet → Express (webhook) | Webhook do Digisac chega via HTTPS; validação de token já existe (WBHK-02) mas agora pode ser chamado de qualquer origem pública no Railway |
| Env vars (Railway dashboard) → processo | SANDBOX_NUMBERS contém contactIds reais — não são segredos mas expõem quais números estão sendo testados |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | `/health` endpoint | accept | Retorna apenas `status` e `uptime` — sem info de versão, env, dependências ou estado interno. Baixo valor para atacante. D-02 explicita payload mínimo; campos adicionais (MON-01) são v2 deferred. |
| T-04-02 | Denial of Service | `/health` endpoint (fora do rate-limit) | mitigate | Endpoint é O(1) sem I/O (apenas `process.uptime()`); custo computacional desprezível. Helmet aplicado ANTES da rota mantém headers de segurança. |
| T-04-03 | Spoofing | Guard 0 sandbox | mitigate | `contactId` vem do payload do Digisac — que já é autenticado via token (WBHK-02) antes do Guard 0 executar. O Set de `sandboxNumbers` é criado no module-load, não vulnerável a injection em runtime. |
| T-04-04 | Elevation of Privilege | SANDBOX_MODE bypass | mitigate | SANDBOX_MODE é env var — só o operador do Railway dashboard pode mudar. Default é `false`. Bypass só acontece se credentials do Railway forem comprometidas (fora do escopo da app). |
| T-04-05 | Information Disclosure | Logs `event:sandbox_blocked` com contactId | accept | contactId é pseudo-anônimo (UUID Digisac), não PII direta. Log em debug-level (não info). Risco baixo; facilita debug operacional durante sandbox (D-11 valor principal). |
| T-04-06 | Tampering | `app.listen` bind em 0.0.0.0 | accept | Bind em 0.0.0.0 dentro do container Railway expõe apenas à rede interna do Railway; porta externa só é exposta via HTTPS com TLS do Railway. Sem downgrade para HTTP. |
</threat_model>
